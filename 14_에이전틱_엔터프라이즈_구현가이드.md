## 전체 기술 트리 (Tech Tree)

```
Level 0: 기초 역량
├── Python 프로그래밍
├── API 연동 (REST, Webhook)
├── 프롬프트 엔지니어링
└── LLM 기본 이해 (토큰, 컨텍스트 윈도우, 할루시네이션)

Level 1: 단일 에이전트
├── LangChain / LangGraph
├── Tool Calling (Function Calling)
├── RAG (검색 증강 생성)
└── 메모리 관리 (단기/장기)

Level 2: 멀티 에이전트
├── 오케스트레이션 패턴 (순차/병렬/라우터)
├── 에이전트 간 통신 프로토콜 (A2A)
├── MCP (Model Context Protocol)
└── 상태 관리 & 에러 핸들링

Level 3: 엔터프라이즈 통합
├── 워크플로우 자동화 (n8n / Make)
├── 기존 시스템 연동 (ERP, CRM, 그룹웨어)
├── 보안 & 거버넌스
└── 모니터링 & 관측성 (Observability)

Level 4: 자율 운영
├── 자기 평가 & 개선 루프
├── Human-in-the-Loop 설계
├── 조직 변화 관리
└── ROI 측정 & 최적화
```

## Phase 1: 단일 에이전트 구축 (0~4주)

> **목표**: "하나의 업무를 대신하는 AI 에이전트" 1개 만들기

### 필수 기술 스택

| 기술 | 용도 | 학습 우선순위 |
| --- | --- | --- |
| **Python** | 에이전트 개발 언어 | ★★★ 필수 |
| **LangChain** | 에이전트 프레임워크 | ★★★ 필수 |
| **OpenAI API / Gemini API** | LLM 호출 | ★★★ 필수 |
| **프롬프트 엔지니어링** | 에이전트 행동 설계 | ★★★ 필수 |

### 핵심 개념: Tool Calling

```
[사용자 질문]
      ↓
[에이전트 (LLM)]  ← "이 질문에 답하려면 어떤 도구를 써야 할까?"
      ↓
[도구 선택] → DB 조회 / API 호출 / 계산 / 파일 읽기
      ↓
[도구 실행 결과]
      ↓
[에이전트] → 결과를 해석하여 최종 답변 생성
```

에이전트는 단순 챗봇과 달리, **스스로 "어떤 도구를 써야 하는지" 판단**합니다.

### 실전 예시: 고객 문의 자동 응답 에이전트

```python
# 개념적 구조 (LangChain 기반)

도구(Tools):
├── search_faq(query)        # FAQ DB에서 관련 답변 검색 (RAG)
├── check_order(order_id)    # 주문 상태 조회 API
├── create_ticket(content)   # 고객 지원 티켓 생성
└── escalate_to_human(reason) # 사람에게 넘기기

에이전트 동작 흐름:
1. 고객: "주문번호 12345 배송이 어디쯤 왔나요?"
2. 에이전트 판단: → check_order 도구 사용
3. 도구 실행: 주문 DB에서 배송 상태 조회
4. 결과 해석: "현재 배송 중이며 내일 도착 예정"
5. 답변 생성: 자연어로 고객에게 안내
```

### Quick Win 파일럿 후보 (난이도순)

| # | 유스케이스 | 난이도 | 예상 효과 |
| --- | --- | --- | --- |
| 1 | **사내 FAQ 챗봇** (RAG 기반) | ⭐ | 반복 질문 70% 자동 응답 |
| 2 | **회의록 자동 요약** | ⭐ | 회의 후처리 시간 80% 감소 |
| 3 | **이메일 초안 작성** | ⭐⭐ | 이메일 작성 시간 50% 단축 |
| 4 | **고객 문의 분류 & 라우팅** | ⭐⭐ | 1차 응답 시간 90% 단축 |
| 5 | **문서 자동 분류 & 태깅** | ⭐⭐ | 문서 관리 시간 60% 감소 |

## Phase 2: RAG + 메모리로 "똑똑한" 에이전트 만들기 (4~8주)

> **목표**: 에이전트가 회사 내부 지식을 활용하고, 대화 맥락을 기억하게 만들기

### 추가 기술 스택

| 기술 | 용도 | 학습 우선순위 |
| --- | --- | --- |
| **벡터 DB** (Chroma, Qdrant) | 문서 임베딩 저장 & 검색 | ★★★ 필수 |
| **임베딩 모델** | 텍스트 → 벡터 변환 | ★★★ 필수 |
| **청킹 전략** | 문서를 적절한 크기로 분할 | ★★ 중요 |
| **LangGraph** | 복잡한 에이전트 워크플로우 | ★★ 중요 |

### RAG 에이전트 아키텍처

```
[회사 내부 문서]
  ├── 매뉴얼, 규정집, FAQ
  ├── 이전 보고서, 회의록
  └── 고객 데이터 (비식별화)
        ↓
[문서 처리 파이프라인]
  ├── 텍스트 추출 (PDF, DOCX, Notion)
  ├── 청킹 (시맨틱 / 단락 기반)
  └── 임베딩 생성 & 벡터 DB 저장
        ↓
[RAG 에이전트]
  ├── 사용자 질문 → 쿼리 변환
  ├── 벡터 유사도 검색 (Top-K)
  ├── 관련 문서 + 질문 → LLM
  └── 답변 + 출처 표기
```

### 메모리 시스템 설계

| 메모리 유형 | 저장 내용 | 수명 | 구현 방법 |
| --- | --- | --- | --- |
| **작업 메모리** | 현재 대화 맥락 | 세션 중 | LangChain ConversationBuffer |
| **단기 메모리** | 최근 N턴 대화 | 수 시간 | 슬라이딩 윈도우 |
| **장기 메모리** | 사용자 선호, 과거 결정 | 영구 | 벡터 DB + 요약 저장 |

> 상세 RAG 설계는 [13_RAG_파일럿_설계서.md](./13_RAG_파일럿_설계서.md) 참조

## Phase 3: 멀티 에이전트 시스템 (8~16주)

> **목표**: 여러 에이전트가 팀처럼 협업하는 시스템 구축

### 추가 기술 스택

| 기술 | 용도 | 학습 우선순위 |
| --- | --- | --- |
| **LangGraph** (고급) | 멀티 에이전트 오케스트레이션 | ★★★ 필수 |
| **A2A 프로토콜** | 에이전트 간 통신 표준 | ★★ 중요 |
| **MCP** | AI ↔ 외부 도구 연결 | ★★ 중요 |
| **상태 머신 설계** | 워크플로우 상태 관리 | ★★ 중요 |

### 4가지 오케스트레이션 패턴

#### 패턴 1: 순차형 (Sequential)
```
[에이전트 A] → [에이전트 B] → [에이전트 C] → 최종 결과
  리서치         분석           보고서 작성
```
**적합**: 문서 작성 파이프라인, 데이터 처리 흐름

#### 패턴 2: 병렬형 (Parallel)
```
           ┌→ [에이전트 A: 시장 조사] ─┐
[입력] ────┼→ [에이전트 B: 경쟁 분석] ─┼→ [종합 에이전트] → 결과
           └→ [에이전트 C: 내부 데이터]─┘
```
**적합**: 여러 소스에서 동시에 정보 수집

#### 패턴 3: 라우터형 (Router)
```
              ┌→ [HR 에이전트]    (인사 관련)
[입력] → [라우터] → [IT 에이전트]    (기술 관련)
              └→ [재무 에이전트]   (비용 관련)
```
**적합**: 고객 문의 분류, 부서별 업무 자동 배분

#### 패턴 4: 감독자형 (Supervisor)
```
[감독자 에이전트]
  ├── 작업 분배 → [실행 에이전트 1]
  ├── 품질 검증 → [실행 에이전트 2]
  ├── 재작업 지시 (품질 미달 시)
  └── 최종 승인
```
**적합**: 보고서 작성, 코드 리뷰, 품질 관리가 필요한 작업

### 실전 예시: 주간 보고서 자동 생성 시스템

```
[감독자 에이전트] ──── "이번 주 영업 보고서를 만들어라"
      │
      ├──→ [데이터 수집 에이전트]
      │      ├── CRM에서 이번 주 거래 데이터 조회
      │      ├── 이메일에서 고객 피드백 추출
      │      └── 결과를 감독자에게 전달
      │
      ├──→ [분석 에이전트]
      │      ├── 전주 대비 실적 비교
      │      ├── 주요 이슈 도출
      │      └── 분석 결과를 감독자에게 전달
      │
      ├──→ [보고서 작성 에이전트]
      │      ├── 데이터 + 분석 → 보고서 초안 작성
      │      └── 초안을 감독자에게 전달
      │
      └──→ [감독자] → 품질 검토 → 승인 or 수정 지시
                         ↓
                    [최종 보고서 발송]
```

## AX 컨설턴트로서의 포지셔닝

이 기술 트리를 모두 직접 마스터할 필요는 없습니다. **AX 아키텍트**로서의 핵심 가치는:

1. **설계**: "어떤 프로세스에 에이전트를 위임할 것인가?" → AXEL Phase 2
2. **구현**: Quick Win 파일럿을 직접 만들 수 있는 실행력 → AXEL Phase 3
3. **확산**: 조직 변화 관리 + 거버넌스 수립 → AXEL Phase 4

> **"AI 에이전트를 만드는 사람은 많다. 하지만 '어떤 업무를 에이전트에게 맡겨야 하는지' 설계하고, 조직이 그 변화를 받아들이도록 이끄는 사람은 드물다."**
