## 1. 프로젝트 개요

### 왜 RAG인가?

기업이 AI를 도입할 때 가장 즉각적인 가치를 만들 수 있는 유스케이스는
**"우리 회사 내부 문서를 AI가 이해하고 검색·요약해주는 시스템"**입니다.

| 기존 방식 | RAG 방식 |
| --- | --- |
| 공유 폴더에서 파일 검색 | 자연어 질문으로 답변 |
| 파일명 기반 추측 탐색 | 의미 기반 관련 문서 자동 연결 |
| 읽고 해석은 사람이 직접 | AI가 핵심을 요약해서 제시 |
| 신규 입사자: 학습 기간 수 주 | 첫 날부터 AI에게 질문 가능 |

### 파일럿 프로젝트 기획서

```

│ 목적      │ 직원들이 회사 내부 문서를 자연어로          │
```

## 2. RAG 아키텍처 설계

### 전체 파이프라인

```
[문서 수집]    [문서 처리]     [벡터 저장]     [검색 & 생성]
    │             │              │               │
    ▼             ▼              ▼               ▼
 PDF/DOCX    청킹(Chunking)   임베딩 생성      사용자 질문
 Markdown    메타데이터 추출   벡터 DB 저장     ↓
 Notion      텍스트 정제       인덱스 구축      쿼리 변환
 Confluence                                    ↓
                                              벡터 유사도 검색
                                               ↓
                                              관련 청크 검색 (Top-K)
                                               ↓
                                              리랭킹 (선택)
                                               ↓
                                              컨텍스트 + 질문 → LLM
                                               ↓
                                              답변 + 출처 표기
```

## 3. 기술 선택 가이드

### 3-1. 청킹(Chunking) 전략 선택

> rag-architect 스킬의 6가지 청킹 전략을 기업 문서 유형별로 매핑

| 문서 유형 | 권장 청킹 | 청크 크기 | 오버랩 | 이유 |
| --- | --- | --- | --- | --- |
| 업무 매뉴얼 | **시맨틱 청킹** | 500~1000자 | 15% | 섹션별 주제 일관성 유지 |
| FAQ 문서 | **단락 기반** | Q&A 단위 | 0% | 질문-답변 쌍 보존 |
| 규정집 | **재귀적 청킹** | 800자 | 10% | 조항 단위 분할, 맥락 유지 |
| 회의록 | **문장 기반** | 300~500자 | 20% | 대화 맥락 연결 |
| 기술 문서 | **문서 인식형** | 1000자 | 10% | 표·코드 블록 보존 |

### 3-2. 임베딩 모델 선택

| 모델 | 차원 | 속도 | 한국어 | 비용 | 권장 사용처 |
| --- | --- | --- | --- | --- | --- |
| `text-embedding-3-small` | 1536 | 빠름 | ⭐⭐ | 저렴 | 프로토타입, 소규모 |
| `text-embedding-3-large` | 3072 | 보통 | ⭐⭐ | 보통 | 프로덕션 |
| `multilingual-e5-large` | 1024 | 보통 | ⭐⭐⭐ | 무료(오픈) | **한국어 최적화** |
| `ko-sroberta-multitask` | 768 | 빠름 | ⭐⭐⭐ | 무료(오픈) | 한국어 특화 |

> 💡 **권장**: 한국어 문서 중심이라면 `multilingual-e5-large` 또는 `ko-sroberta` 우선 테스트

### 3-3. 벡터 데이터베이스 선택

| DB | 유형 | 장점 | 단점 | 권장 사용처 |
| --- | --- | --- | --- | --- |
| **Chroma** | 임베디드 | 빠른 프로토타이핑, 쉬운 설정 | 스케일 한계 | ★ PoC / 파일럿 |
| **Qdrant** | 자체 호스팅 | 고성능, 낮은 메모리 | 인프라 관리 필요 | 프로덕션 (자체 서버) |
| **pgvector** | PostgreSQL 확장 | 기존 DB 활용, SQL 조인 | 전문 벡터DB 대비 성능↓ | 기존 PostgreSQL 환경 |
| **Pinecone** | 관리형 SaaS | 풀 매니지드, 자동 스케일 | 비용, 벤더 종속 | 프로덕션 (클라우드) |

> 💡 **파일럿 권장**: Chroma로 시작 → 프로덕션 시 Qdrant 또는 pgvector로 마이그레이션

### 3-4. 검색 전략

| 전략 | 설명 | 정확도 | 복잡도 | 권장 |
| --- | --- | --- | --- | --- |
| **Dense만** | 임베딩 유사도 검색 | ⭐⭐⭐ | 낮음 | 파일럿 시작 |
| **Hybrid** | Dense + BM25(키워드) | ⭐⭐⭐⭐ | 중간 | ★ 프로덕션 |
| **Hybrid + 리랭킹** | 2단계 검색 | ⭐⭐⭐⭐⭐ | 높음 | 높은 정확도 필요 시 |

## 4. 8주 실행 타임라인

```
Week 1~2: 데이터 수집 & 정제
├── 대상 문서 수집 및 현황 파악
├── 문서 포맷 통일 (PDF→텍스트, 표 추출)
├── 메타데이터 태깅 (부서, 작성일, 유형)
└── 데이터 품질 기준 수립

Week 3~4: 파이프라인 구축
├── 청킹 전략 테스트 (2~3가지 비교)
├── 임베딩 모델 벤치마크
├── 벡터 DB 설정 및 인덱싱
└── 기본 검색 기능 구현

Week 5~6: RAG 시스템 통합
├── LLM 연동 (프롬프트 엔지니어링)
├── 출처 표기 시스템 구현
├── 할루시네이션 방지 가드레일 적용
├── UI/챗봇 인터페이스 구축
└── 내부 테스트 (10명)

Week 7~8: 평가 & 최적화
├── 평가 데이터셋으로 정량 테스트
├── 사용자 피드백 수집 & 반영
├── 성능 최적화 (캐싱, 리랭킹)
└── 파일럿 결과 보고서 작성
```

## 5. 평가 프레임워크

### 정량 지표 (RAGAS 기반)

| 지표 | 설명 | 목표 | 측정 방법 |
| --- | --- | --- | --- |
| **Faithfulness** | 답변이 검색된 문서에 근거하는가? | > 90% | NLI 모델로 검증 |
| **Context Relevance** | 검색된 청크가 질문과 관련 있는가? | > 80% | 임베딩 유사도 |
| **Answer Relevance** | 답변이 질문에 적합한가? | > 85% | LLM-as-Judge |
| **Precision@5** | 상위 5개 결과 중 관련 있는 비율 | > 70% | 사람 평가 |

### 정성 지표

| 지표 | 측정 방법 | 목표 |
| --- | --- | --- |
| 사용자 만족도 | 5점 척도 설문 | > 4.0/5.0 |
| 정보 탐색 시간 | Before/After 시간 측정 | 50% 이상 단축 |
| 재사용 의향 | "다시 사용하겠는가?" | > 80% "예" |

## 6. 보안 & 거버넌스 가드레일

### 데이터 보안

- [ ] 기밀 문서 등급 분류 (공개/내부/기밀) — 등급별 접근 범위 설정
- [ ] 개인정보(PII) 자동 탐지 및 비식별화 처리
- [ ] 데이터 외부 전송 여부 결정 (클라우드 API vs 온프레미스)
- [ ] 임베딩 모델의 데이터 학습 방지 설정 확인

### AI 출력 안전성

- [ ] 할루시네이션 감지: 출처 없는 답변 자동 경고
- [ ] Human-in-the-loop: 주요 의사결정 시 사람 승인 필수
- [ ] 답변 신뢰도 점수 표시 (높음/중간/낮음)
- [ ] "모르겠습니다" 응답 — AI가 모를 때 지어내지 않도록 설정

### 접근 제어

- [ ] 부서별 문서 접근 권한 → RAG 검색 결과에도 동일 적용
- [ ] 사용 로그 기록 (누가, 언제, 무엇을 질문했는가)
- [ ] 관리자 대시보드 (사용량, 오류율, 피드백)

## 8. 파일럿 → 프로덕션 확장 경로

```
[파일럿 성공]
    │
    ├─→ 1. 문서 범위 확대 (전 부서 문서 인덱싱)
    │
    ├─→ 2. 고급 기능 추가
    │   ├── Hybrid Search (Dense + BM25)
    │   ├── 리랭킹 모델 도입
    │   ├── 다국어 지원 (영어 문서 포함)
    │   └── 슬랙/팀즈 연동 챗봇
    │
    ├─→ 3. 인프라 스케일업
    │   ├── Chroma → Qdrant/pgvector 마이그레이션
    │   ├── 캐싱 레이어 추가
    │   └── 모니터링 & 알림 시스템
    │
    └─→ 4. 거버넌스 고도화
        ├── 문서 자동 갱신 파이프라인
        ├── 정기 성능 평가 (월 1회)
        └── 사용자 피드백 → 자동 개선 루프
```
